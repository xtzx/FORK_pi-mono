├────────────────────────────────────────────────────────────┤
│                    Agent 运行时功能                          │
│  Agent 循环 · 工具调用编排 · 状态管理 · 事件发布              │

## 二、LLM 通信层功能（packages/ai）

### 2.1 多 Provider 统一 API

**功能描述**：通过一套统一的 `stream(model, context, options)` API 与 10+ 种 LLM Provider 通信，上层代码完全不需要感知底层 Provider 的协议差异。

**支持的 Provider 列表**：

| Provider | API 类型 | 认证方式 |
|----------|---------|---------|
| Anthropic | anthropic-messages | API Key / OAuth |
| OpenAI | openai-completions / openai-responses | API Key |
| OpenAI Codex | openai-codex-responses | OAuth |
| Azure OpenAI | azure-openai-responses | API Key |
| Google Gemini | google-generative-ai | API Key / OAuth |
| Google Vertex AI | google-vertex | Service Account |
| Google Gemini CLI | google-gemini-cli | OAuth |
| Mistral | mistral-conversations | API Key |
| Amazon Bedrock | bedrock-converse-stream | AWS Credentials |
| GitHub Copilot | anthropic-messages | OAuth |
| xAI (Grok) | openai-completions | API Key |
| Groq | openai-completions | API Key |
| Cerebras | openai-completions | API Key |
| OpenRouter | openai-completions | API Key |
| Vercel AI Gateway | openai-completions | API Key |
| HuggingFace | openai-completions | API Key |
| Minimax | openai-completions | API Key |
| Kimi Coding | openai-completions | API Key |

**兼容性层**：对于 OpenAI 兼容 API（如 xAI、Groq、OpenRouter 等），`OpenAICompletionsCompat` 接口提供了精细的兼容性控制：是否支持 `store` 字段、是否支持 `developer` 角色、是否支持 `reasoning_effort`、max tokens 字段名等。这些参数可以根据 URL 自动检测，也可以手动覆盖。

### 2.2 流式响应处理

**功能描述**：将 LLM 的流式输出标准化为统一的事件序列，支持文本、思考过程、工具调用三种内容类型的增量输出。

**事件生命周期**：
```
start
  ├── text_start → text_delta (×N) → text_end
  ├── thinking_start → thinking_delta (×N) → thinking_end
  └── toolcall_start → toolcall_delta (×N) → toolcall_end
done/error
```

**核心技术**：
- `EventStream<T, R>` 实现了带终止语义的异步迭代器
- 工具调用参数通过 `partial-json` 库在流式过程中进行增量解析
- 每个事件都携带当前的 `partial` AssistantMessage，使得消费者可以在任意时刻获取到最新状态

### 2.3 Token 计数与成本追踪

**功能描述**：精确追踪每次 LLM 调用的 Token 使用量和成本。

**追踪维度**：
- `input`：输入 Token 数
- `output`：输出 Token 数
- `cacheRead`：从缓存读取的 Token 数
- `cacheWrite`：写入缓存的 Token 数
- `totalTokens`：总 Token 数（部分 Provider 提供原生值）

**成本计算**：每个 Model 对象包含 `cost` 字段（$/百万 Token），`calculateCost()` 函数根据 usage 和 cost 计算实际费用。缓存读取和写入有独立的费率。

### 2.4 模型自动发现

**功能描述**：通过脚本从各 Provider 的 API 自动拉取可用模型列表，生成 `models.generated.ts`（13,370 行）。

**数据来源**：
- models.dev API
- OpenRouter 模型列表
- Vercel AI Gateway 模型列表
- 各 Provider 官方 API

**生成的模型信息**包含：ID、名称、Provider、API 类型、输入能力（文本/图片）、推理能力、上下文窗口大小、最大输出 Token 数、成本信息等。

### 2.5 工具调用参数校验

**功能描述**：使用 TypeBox 定义工具参数 Schema，通过 AJV 在运行时校验 LLM 返回的工具调用参数。

**技术要点**：
- TypeBox 提供 TypeScript 层面的类型推导（`Static<typeof schema>` 自动推导参数类型）
- AJV 提供运行时的 JSON Schema 校验
- 校验失败时抛出错误，由 Agent 循环捕获并转为 ToolResultMessage 反馈给 LLM

### 2.6 跨 Provider 消息转换

**功能描述**：`transformMessages()` 函数处理不同 Provider 间的消息格式差异，使得同一个对话历史可以在 Provider 之间无缝切换。

**处理内容**：
- 思考块（thinking）的跨 Provider 兼容性：部分 Provider 不支持思考块，需要转为文本或移除
- 工具调用 ID 格式差异：不同 Provider 对 tool call ID 有不同的格式要求
- 消息角色映射：部分 Provider 用 `developer` 替代 `system`
- 签名字段处理：OpenAI 的 `textSignature`、Google 的 `thoughtSignature` 等不能跨 Provider 传递

### 2.7 上下文溢出检测

**功能描述**：检测 LLM 返回的错误是否为上下文溢出（即消息历史超过了模型的上下文窗口限制）。

**实现方式**：`isContextOverflow()` 函数通过模式匹配错误消息文本来识别各 Provider 的上下文溢出错误格式。支持 Anthropic、OpenAI、Google、Mistral、Bedrock 等 Provider 的错误格式。

### 2.8 OAuth 认证

**功能描述**：完整的 OAuth 流程支持，用于那些不提供简单 API Key 的 Provider。

**支持的 OAuth 流程**：
- Anthropic Console OAuth
- OpenAI Codex OAuth
- GitHub Copilot Device Flow + Token Refresh
- Google Gemini CLI OAuth（使用 Google Cloud 的 antigravity CLI 协议）
- PKCE（Proof Key for Code Exchange）验证

**Token 管理**：自动刷新过期 Token，存储在 `auth.json` 中。对于 GitHub Copilot 等短期 Token，Agent Loop 每次 LLM 调用前都通过 `getApiKey()` 回调动态获取最新 Token。

## 三、Agent 运行时功能（packages/agent）

### 3.1 Agent 循环（Agent Loop）

**功能描述**：实现 "提示 → LLM 推理 → 工具执行 → LLM 推理 → ..." 的自动化循环，直到 LLM 认为任务完成或被中断。

**双层循环架构**：

外层循环处理 follow-up 消息——当 Agent "认为完成"后，如果有排队的 follow-up 消息，将其注入上下文并继续。

内层循环处理工具调用和 steering 消息——LLM 返回工具调用后，依次执行工具，每个工具执行后检查 steering 消息。如果有 steering，跳过剩余工具并将 steering 消息注入上下文。

**两个入口**：
- `agentLoop(prompts, context, config)`: 新 prompt，从头开始
- `agentLoopContinue(context, config)`: 从已有上下文继续（用于重试）

### 3.2 工具调用编排

**功能描述**：从 LLM 的 AssistantMessage 中提取工具调用，依次执行，收集结果。

**执行流程**：
1. 提取所有 `type === "toolCall"` 的内容块
2. 对每个工具调用：
   - 发布 `tool_execution_start` 事件
   - 用 `validateToolArguments()` 校验参数
   - 调用 `tool.execute(id, args, signal, onUpdate)` 执行
   - 执行过程中通过 `onUpdate` 回调流式更新进度
   - 捕获错误，构造 `isError: true` 的结果
   - 发布 `tool_execution_end` 事件
3. 每个工具执行后检查 steering 消息
4. 如果有 steering 消息，跳过剩余工具（为被跳过的工具生成 "Skipped" 结果）

### 3.3 状态管理

**功能描述**：Agent 类维护完整的对话状态，包括消息历史、当前模型、工具集、流式状态等。

**AgentState 包含**：
- `systemPrompt`: 系统提示
- `model`: 当前模型
- `thinkingLevel`: 思考级别（off/minimal/low/medium/high/xhigh）
- `tools`: 当前可用工具列表
- `messages`: 消息历史（AgentMessage[]）
- `isStreaming`: 是否正在流式响应
- `streamMessage`: 当前流式消息（实时更新的 partial message）
- `pendingToolCalls`: 正在执行的工具调用 ID 集合
- `error`: 错误信息

**不可变更新**：消息列表的修改都通过创建新数组实现（`[...messages, newMsg]`、`messages.slice()`），避免了引用共享导致的状态不一致。

### 3.4 事件发布系统

**功能描述**：Agent 通过事件系统向外部通知状态变化，实现 Agent 循环与 UI 的解耦。

**事件类型**：
- `agent_start` / `agent_end`: Agent 生命周期
- `turn_start` / `turn_end`: 每一轮 LLM 调用的生命周期
- `message_start` / `message_update` / `message_end`: 消息级别的生命周期
- `tool_execution_start` / `tool_execution_update` / `tool_execution_end`: 工具执行生命周期

**订阅模式**：`agent.subscribe(fn)` 返回取消订阅函数，典型的发布-订阅模式。

### 3.5 Steering（中途引导）

**功能描述**：允许在 Agent 运行过程中注入消息来改变 Agent 的行为方向。

**使用场景**：
- 用户在 Agent 执行工具时输入了新的指令
- 扩展检测到某种条件并需要引导 Agent 转向
- 需要立即纠正 Agent 的错误行为

**两种模式**：
- `"all"`：一次性发送所有排队的 steering 消息
- `"one-at-a-time"`：每次只发送一条，确保 Agent 逐一处理

### 3.6 Follow-up（后续追加）

**功能描述**：在 Agent 完成当前任务后自动追加新任务。

**使用场景**：
- 用户在 Agent 忙碌时排队了新消息
- 扩展需要在某个步骤完成后自动触发下一步
- 批处理场景

**与 Steering 的区别**：Steering 会打断当前执行（跳过剩余工具），Follow-up 会等当前任务完全结束后才生效。

### 3.7 代理转发（Proxy）

**功能描述**：`streamProxy()` 函数允许将 LLM 请求转发到代理服务器，实现负载均衡、审计、缓存等功能。

## 四、编码 Agent 功能（packages/coding-agent）

### 4.1 交互式终端界面

**功能描述**：全屏终端交互界面，支持语法高亮的 Markdown 渲染、工具执行进度显示、多行输入编辑等。

**组件清单**（30+ 个 TUI 组件）：
- 消息列表与流式渲染
- Markdown 渲染（代码高亮、表格、链接等）
- 工具执行详情面板（diff 预览、bash 输出滚动等）
- 模型选择器
- 会话选择器
- 文件路径自动补全
- 斜杠命令自动补全
- 树形会话导航器
- 设置编辑器
- 思考过程折叠显示
- 上下文使用量条形图
- 键绑定帮助面板

**主题系统**：支持 JSON 格式的主题定义，内置深色和浅色主题。主题控制所有 UI 元素的颜色。

### 4.2 文件操作工具集

#### 4.2.1 read 工具

- 读取文件内容，支持 `offset` 和 `limit` 参数（按行）
- 图片文件自动 resize 到合理尺寸后转为 base64
- 超长文件自动截断并提示 LLM 使用 offset/limit 读取特定范围
- 二进制文件检测并拒绝读取

#### 4.2.2 write 工具

- 创建或覆盖文件
- 自动创建中间目录
- 写入后返回确认信息

#### 4.2.3 edit 工具

- **精确文本替换**：`oldText → newText`
- **模糊匹配**：先尝试精确匹配，失败后进行 Unicode 规范化、引号标准化、连字符标准化、空白标准化后重试
- **多重匹配检测**：如果 oldText 在文件中出现多次，报错要求 LLM 提供更多上下文使其唯一
- **BOM 保留**：识别并保留文件的 BOM（Byte Order Mark）
- **行尾符保留**：检测文件原始行尾符（CRLF/LF），替换后恢复
- **Diff 生成**：替换后生成 unified diff 用于 UI 展示

#### 4.2.4 bash 工具

- 在当前工作目录执行 bash 命令
- 支持超时设置
- 输出截断：保留最后 N 行或 M 字节，超出部分写入临时文件
- 流式输出更新：通过 `onUpdate` 回调实时推送命令输出
- 进程树管理：超时或中止时 kill 整个进程树
- 支持 `commandPrefix`（如 `shopt -s expand_aliases`）和 `spawnHook`（修改命令/环境）

#### 4.2.5 grep 工具

- 封装 ripgrep（rg）
- 支持正则匹配、上下文行数、结果限制、字面量模式
- 自动尊重 .gitignore
- 比 bash + grep 更快更安全

#### 4.2.6 find 工具

- 封装 fd（find alternative）
- Glob 模式文件搜索
- 自动尊重 .gitignore
- 比 bash + find 更快更安全

#### 4.2.7 ls 工具

- 列出目录内容
- 目录名带 `/` 后缀以区分文件
- 输出截断防止超大目录消耗过多 Token

### 4.3 会话管理

**功能描述**：完整的对话会话生命周期管理，支持持久化、恢复、分支、压缩。

#### 4.3.1 JSONL 持久化

每个会话是一个 JSONL 文件，每行是一个 JSON 对象（条目）。条目类型包括：
- `session`: 会话头信息（ID、创建时间、工作目录）
- `message`: 消息（user/assistant/toolResult/custom）
- `thinking_level_change`: 思考级别变更
- `model_change`: 模型变更
- `compaction`: 上下文压缩记录
- `branch_summary`: 分支摘要
- `custom`: 扩展自定义数据
- `custom_message`: 扩展自定义消息
- `label`: 会话标签
- `session_info`: 会话元数据（描述等）

#### 4.3.2 树形结构

每个条目有 `id`（UUID）和 `parentId`，形成有向无环图（实际是树）。分支通过创建新条目并指向同一个父节点实现。从任意叶节点回溯到根节点即可重建完整的会话路径。

#### 4.3.3 会话恢复

`--resume` 命令可以恢复之前的会话。`SessionPicker` TUI 组件显示所有历史会话供用户选择。恢复时从 JSONL 文件重建完整的消息历史和设置状态。

#### 4.3.4 会话分支（Fork）

在任意消息处创建分支，新分支共享分支点之前的所有历史。分支点后的内容独立演进。分支前可以生成分支摘要注入到新分支中，让 LLM 知道"这里从另一个分支的结果开始"。

### 4.4 上下文压缩（Compaction）

**功能描述**：当对话历史接近模型的上下文窗口限制时，自动或手动将旧消息总结为简洁的摘要。

**触发条件**：
- 自动：`estimatedTokens > contextWindow - reserveTokens`（默认保留 16,384 Token 余量）
- 手动：`/compact` 命令
- 上下文溢出后自动重试

**压缩算法**：
1. 从最新条目向前扫描，累积 Token 估算值
2. 当累积到 `keepRecentTokens`（默认 20,000）时确定切割点
3. 切割点前的消息交给 LLM 生成结构化摘要
4. 摘要格式：Goal、Constraints、Progress（Done/In Progress/Blocked）、Key Decisions、Next Steps、Critical Context
5. 如果切割点在一个 turn（用户消息→助手回复→工具结果）中间，还会生成 turn prefix 摘要
6. 追踪文件操作（已读文件、已修改文件列表）附加到摘要中

**迭代更新**：如果之前已经有一次压缩，新的压缩会使用"更新摘要"提示，将新消息的变化合并到已有摘要中，而非从头生成。

### 4.5 系统提示构建

**功能描述**：动态构建发给 LLM 的系统提示，整合工具描述、指南、项目上下文、技能等。

**系统提示的组成部分**：
1. 角色定义："You are an expert coding assistant operating inside pi..."
2. 可用工具列表（根据实际启用的工具动态生成）
3. 使用指南（根据启用的工具动态生成相关指南）
4. Pi 文档路径（仅在用户问到 Pi 本身时使用）
5. 追加内容（`appendSystemPrompt`）
6. 项目上下文（AGENTS.md / CLAUDE.md 内容）
7. 技能列表（可用的 Agent Skills）
8. 当前日期时间
9. 当前工作目录

**可覆盖性**：
- 完全替换：`.pi/SYSTEM.md` 文件可替换整个默认提示
- 追加：`APPEND_SYSTEM.md` 可在默认提示后追加内容
- 项目上下文：自动从 CWD 向上查找 `AGENTS.md` 或 `CLAUDE.md`

### 4.6 模型管理

**功能描述**：支持在多个模型和 Provider 之间切换，包括自定义模型配置。

**模型解析**：
- CLI 参数：`--provider anthropic --model claude-sonnet-4-20250514`
- 简写：`anthropic/claude-sonnet-4-20250514`
- 带思考级别：`model:high`
- Glob 作用域：`--models "anthropic/*" "google/gemini-*"`

**模型注册**：
- 内置模型来自 `models.generated.ts`
- 自定义模型通过 `~/.pi/agent/models.json` 添加
- 扩展可通过 `pi.registerProvider()` 动态注册新 Provider 和模型

**运行时切换**：快捷键循环切换当前可用模型（在 scoped models 或全部模型中循环）。切换时自动处理跨 Provider 的消息格式兼容性。

### 4.7 扩展系统

**功能描述**：TypeScript 扩展可以在运行时加载，扩展 Pi 的功能。

**扩展能力清单**：

| 能力 | API | 说明 |
|------|-----|------|
| 注册工具 | `pi.registerTool()` | 添加新的 Agent 工具 |
| 注册命令 | `pi.registerCommand()` | 添加新的斜杠命令 |
| 注册快捷键 | `pi.registerShortcut()` | 添加新的键绑定 |
| 注册 CLI Flag | `pi.registerFlag()` | 添加新的 CLI 参数 |
| 注册 Provider | `pi.registerProvider()` | 添加新的 LLM Provider |
| 注册消息渲染器 | `pi.registerMessageRenderer()` | 自定义消息在 TUI 中的渲染方式 |
| 监听事件 | `pi.on(event, handler)` | 响应 Agent 生命周期事件 |
| UI 操作 | `ctx.ui.*` | 弹窗、输入、选择、通知等 |
| 拦截工具调用 | `tool_call` 事件 | 在工具执行前阻止或修改 |
| 拦截工具结果 | `tool_result` 事件 | 在结果返回 LLM 前修改 |
| 压缩钩子 | `session_before_compact` | 自定义压缩逻辑 |
| 分支钩子 | `session_before_fork` | 分支前的自定义操作 |

**加载机制**：
- 本地扩展：通过 `--extension path/to/extension.ts` 加载
- Pi 包：通过 `pi install npm:@foo/bar` 安装到 `~/.pi/packages/`
- 使用 jiti（Just-In-Time TypeScript transpiler）运行时转译 TypeScript

**事件系统**：扩展通过 `ExtensionRunner` 参与事件分发。事件流经所有注册的扩展，每个扩展可以：
- 观察事件（logging、metrics）
- 修改事件数据
- 阻止事件传播（如阻止危险命令执行）
- 触发副作用（如发通知、写日志）

### 4.8 斜杠命令系统

**功能描述**：以 `/` 开头的命令提供快捷操作，不发送给 LLM。

**内置命令**：

| 命令 | 功能 |
|------|------|
| `/help` | 显示帮助信息 |
| `/model` | 切换模型 |
| `/thinking` | 切换思考级别 |
| `/compact` | 手动触发上下文压缩 |
| `/clear` | 清除消息历史 |
| `/session` | 显示会话统计 |
| `/export` | 导出会话为 HTML |
| `/login` | OAuth 登录 |
| `/logout` | 清除 OAuth Token |
| `/config` | 打开配置编辑器 |
| `/bug` | 生成 bug 报告 |
| `/tree` | 打开会话树导航器 |
| `/fork` | 分支当前会话 |
| `/skill:name` | 调用指定技能 |
| `/template_name` | 展开 Prompt 模板 |

### 4.9 Prompt 模板系统

**功能描述**：预定义的 Markdown 文件作为 prompt 模板，使用 `/name` 展开。

**模板查找路径**：
- 项目级：`.pi/prompts/`
- 全局级：`~/.pi/agent/prompts/`

**Frontmatter 支持**：模板文件可以包含 YAML frontmatter，定义描述、参数等元数据。

### 4.10 技能系统（Agent Skills）

**功能描述**：基于 `SKILL.md` 标准的可复用 Agent 能力单元。

**工作方式**：
1. 技能定义在 `SKILL.md` 文件中
2. 系统提示中列出可用技能的名称和描述
3. 用户通过 `/skill:name` 或自然语言触发
4. Pi 读取 SKILL.md 内容并注入到对话中
5. LLM 根据技能描述执行相应操作

**技能查找路径**：
- 项目级：`.pi/skills/`
- 全局级：`~/.pi/agent/skills/`
- 扩展包中的技能

### 4.11 权限与安全控制

**功能描述**：通过扩展机制实现灵活的权限控制。

**工作方式**：
- 核心不内置任何命令黑名单或白名单
- 安全策略通过扩展的 `tool_call` 事件实现
- 扩展可以在工具执行前：
  - 检查命令内容（正则匹配危险模式）
  - 弹出确认对话框
  - 直接阻止执行并返回错误

**示例策略**（permission-gate 扩展）：
- 匹配 `rm -rf`、`sudo`、`chmod/chown 777` 等危险模式
- 有 UI 时弹窗确认，无 UI 时直接阻止

### 4.12 HTML 导出

**功能描述**：将完整的会话历史导出为独立的 HTML 文件。

**导出内容**：
- 消息内容（Markdown 渲染为 HTML）
- 工具调用和结果
- 思考过程
- ANSI 颜色代码转换为 HTML 样式
- 自定义模板（template.html + template.css + template.js）

### 4.13 RPC 模式

**功能描述**：通过 stdin/stdout 的 JSON-RPC 协议与外部程序集成。

**支持的操作**：
- 发送 prompt
- 获取当前状态
- 切换模型
- 中止执行
- 事件订阅

**使用场景**：
- 编辑器插件集成（VS Code、Neovim 等）
- 自定义 UI 前端
- 自动化测试

### 4.14 自动重试

**功能描述**：LLM 调用失败时自动重试，支持多种失败场景。

**重试策略**：
- 上下文溢出：先执行压缩，然后重试
- Provider 限速（429/503）：按 Provider 指定的延迟等待后重试
- 一般错误：等待后重试，支持配置最大重试次数和延迟

## 五、终端 UI 功能（packages/tui）

### 5.1 差分渲染引擎

**功能描述**：高效的终端渲染引擎，只更新屏幕上变化的部分。

**三种渲染策略**：
1. 首次渲染：全量写入
2. 尺寸变化：清屏后全量重绘
3. 普通更新：逐行对比，只写入变化行

**同步输出**：使用 CSI 2026（Synchronized Output）协议，确保渲染更新是原子的，避免部分更新导致的闪烁。

### 5.2 Markdown 渲染

**功能描述**：在终端中渲染 Markdown，支持代码高亮、表格、列表、链接等。

**特性**：
- 代码块语法高亮
- 表格对齐渲染
- 链接高亮（可点击，取决于终端支持）
- 列表缩进
- 引用块样式

### 5.3 多行文本编辑器

**功能描述**：支持多行输入的终端文本编辑器组件，约 2,000 行实现。

**特性**：
- 多行编辑
- 光标移动（上下左右、Home/End、Ctrl+方向键）
- 文本选择
- 复制/粘贴（支持 Bracketed Paste）
- 撤销/重做（Undo Stack）
- Kill Ring（Emacs 风格的剪切板）
- 自动换行
- IME 输入支持

### 5.4 自动补全

**功能描述**：斜杠命令和文件路径的自动补全。

**实现**：
- `CombinedAutocompleteProvider` 组合多个补全源
- 斜杠命令补全：匹配已注册的命令
- 文件路径补全：基于 `@` 前缀触发，扫描文件系统
- 模糊匹配算法

### 5.5 内联图片

**功能描述**：在支持的终端中内联显示图片。

**支持的协议**：
- Kitty Graphics Protocol
- iTerm2 Inline Images Protocol

**支持的格式**：PNG、JPEG、GIF、WebP

### 5.6 Overlay 浮动层

**功能描述**：在主内容上方显示浮动面板，用于自动补全下拉框、确认对话框等。

**定位方式**：
- 锚点（相对于某个组件位置）
- 百分比尺寸
- 偏移量
- 可见性回调

### 5.7 键绑定系统

**功能描述**：可配置的键绑定，支持 Kitty 键盘协议的完整修饰键。

**特性**：

- 区分 key press / key repeat / key release
- 编辑器键绑定和应用键绑定分层管理
- 默认键绑定可通过配置文件覆盖

## 十、功能之间的协作关系

### 10.1 一个完整的交互循环

```
用户在 TUI 中输入 "修复 src/app.ts 中的类型错误"
    ↓
InteractiveMode 接收输入
    ↓
AgentSession.prompt() 处理输入
    ├── 检查是否是斜杠命令 → 否
    ├── 展开 Prompt 模板 → 无匹配
    ├── 处理 @file 引用 → 无
    └── 构造 UserMessage
    ↓
Agent.prompt(userMessage)
    ↓
Agent._runLoop()
    ├── 构建 AgentContext
    ├── 构建 AgentLoopConfig（包含 convertToLlm, transformContext 等）
    └── agentLoop(messages, context, config)
        ↓
        runLoop()
            ↓
            streamAssistantResponse()
                ├── transformContext(messages) → 检查是否需要裁剪
                ├── convertToLlm(messages) → 过滤自定义消息类型
                └── streamSimple(model, llmContext, options) → 调用 LLM
                    ↓
                    API 注册表分发到 Anthropic/OpenAI/Google 等 Provider
                    ↓
                    Provider 将请求转为各自 API 格式，发送 HTTP 请求
                    ↓
                    流式解析响应，emit 事件
                        ↓
                        Agent Loop 消费事件
                            ├── message_start → UI 开始渲染
                            ├── message_update → UI 增量更新
                            │   ├── text_delta → 文字逐字显示
                            │   ├── thinking_delta → 思考过程更新
                            │   └── toolcall_delta → 工具调用参数逐步显示
                            └── message_end → UI 完成渲染
            ↓
            检测到 tool calls：[read("src/app.ts"), edit("src/app.ts", ...)]
            ↓
            executeToolCalls()
                ├── tool_execution_start("read") → UI 显示"正在读取文件"
                ├── readTool.execute() → 读取文件内容
                ├── tool_execution_end("read") → UI 显示文件内容
                ├── 检查 steering → 无中断
                ├── tool_execution_start("edit") → UI 显示"正在编辑文件"
                ├── editTool.execute() → 执行模糊匹配替换
                ├── tool_execution_end("edit") → UI 显示 diff
                └── 检查 steering → 无中断
            ↓
            将 tool results 加入 context
            ↓
            streamAssistantResponse() → LLM 确认修复完成
            ↓
            无更多 tool calls → turn_end
            ↓
            检查 follow-up → 无
            ↓
            agent_end
    ↓
AgentSession 将所有新消息持久化到 JSONL
    ↓
InteractiveMode 等待用户下一个输入
```

### 10.2 上下文压缩触发链

```
Agent turn_end 事件
    ↓
AgentSession 检查上下文大小
    ├── estimateContextTokens(messages) → 估算当前 Token 数
    ├── shouldCompact(tokens, contextWindow, settings) → 是否超阈值
    └── 如果需要压缩：
        ↓
        emit auto_compaction_start
        ↓
        prepareCompaction(pathEntries, settings)
            ├── 找到切割点
            ├── 提取待摘要消息
            └── 提取文件操作记录
        ↓
        扩展钩子：session_before_compact → 扩展可自定义压缩
        ↓
        compact(preparation, model, apiKey)
            ├── generateSummary() → 调用 LLM 生成摘要
            └── 附加文件操作列表
        ↓
        SessionManager 保存 compaction 条目
        ↓
        重新加载会话上下文
        ↓
        emit auto_compaction_end
```

## 十一、总结

Pi 实现的功能可以概括为三个核心能力层次：

**基础层**：统一多 Provider LLM 通信、流式处理、Token 追踪、工具参数校验——这些是构建任何 AI Agent 的基础设施。

**运行时层**：Agent 循环编排、状态管理、事件系统、Steering/Follow-up——这些是 Agent 行为控制的核心机制。

**应用层**：文件操作工具、会话管理、扩展系统、TUI/Web UI、上下文压缩——这些是面向最终用户的交互功能。

每一层都可以独立使用，但组合在一起时形成了一个完整的、可扩展的、多 Provider 支持的 AI 编码助手平台。
